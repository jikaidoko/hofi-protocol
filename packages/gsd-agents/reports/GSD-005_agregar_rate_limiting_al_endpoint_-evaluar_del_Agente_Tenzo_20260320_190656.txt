    · [MEDIA] Falta documentación de estrategia de migración si Redis queda inconsistente
    · [BAJA] init_rate_limiter() en main.py falla en startup sin captura granular de errores
    · [BAJA] fakeredis en tests no simula comportamiento de MULTI/EXEC transactions
    · [ALTA] FALTAN HEADERS HTTP DE SEGURIDAD: No hay evidencia en main.py de middleware configurando Security Headers (X-Content-Type-Options, X-Frame-Options, Strict-Transport-Security, Content-Security-Policy). FastAPI no los agrega por defecto.
    · [ALTA] REDIS PASSWORD NO CONFIGURADO: redis_url en .env.example usa redis://localhost:6379/0 sin autenticación. En producción Redis debe requerir password para evitar acceso no autorizado si puerto está expuesto.
    · [ALTA] FALTA VALIDACIÓN DE TAMAÑO MÁXIMO EN PROMPT: agente_tenzo.py valida min_length=1 pero no max_length. Un atacante puede enviar prompt de 10MB causando memory exhaustion o slowloris al procesar.
    · [MEDIA] LOGGING DE BYPASS ATTEMPTS NO IMPLEMENTADO: Nota de seguridad del plan requiere 'loggear attempts de bypass detectados (>10 requests desde misma IP después de 429)' pero no hay evidencia en rate_limiter.py de contador persistente ni log entry cuando check_rate_limit() lanza 429.
    · [MEDIA] FALTA CONFIGURACIÓN CORS: No hay evidencia de CORSMiddleware en main.py. Si frontend está en dominio diferente, requests cross-origin fallarán. Si CORS se agrega sin allow_origins restrictivo, expone API a cualquier origen.
    · [MEDIA] DEPENDENCIAS SIN HASH PINNING: requirements.txt usa versiones específicas (fastapi==0.104.1) pero sin hashes SHA256. Ataque supply-chain puede comprometer PyPI package entre install y deploy.
    · [MEDIA] REDIS CONNECTION POOLING NO CONFIGURADO: redis.from_url() sin max_connections puede causar connection exhaustion bajo carga. Default pool size es ilimitado, permitiendo DoS por agotamiento de file descriptors.
    · [BAJA] FALTA GRACEFUL SHUTDOWN: No hay evidencia de signal handlers (SIGTERM/SIGINT) que cierren Redis connection pool antes de terminar proceso. Puede causar 'broken pipe' errors y dirty shutdown en deploys.
    · [BAJA] ENDPOINT /evaluar NO RETORNA HEADERS X-RateLimit-*: Buena práctica es retornar X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset para que clientes implementen backoff proactivo sin necesidad de recibir 429.
    · [ALTA] TESTS NO VERIFICAN EDGE CASE request.client=None: Criterio C4 nota que 'Client.host puede ser None en conexiones sin IP válida (Unix socket)', pero no hay test test_client_host_none verificando que get_client_ip() retorna 400 en ese caso.
