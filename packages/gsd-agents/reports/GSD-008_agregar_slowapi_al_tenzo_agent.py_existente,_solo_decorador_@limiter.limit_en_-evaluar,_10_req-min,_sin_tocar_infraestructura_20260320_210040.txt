  GSD-008: agregar slowapi al tenzo_agent.py existente, solo decorador @limiter.limit en /evaluar, 10 req/min, sin tocar infraestructura
  Estado      : BLOQUEADO
  Tiempo      : 493.7s
  Reintentos  : 3
══════════════════════════════════════════════════════
  BLOQUEOS DETECTADOS:
    → Vulnerabilidad [CRITICA]: BYPASS DE RATE LIMITING via IP spoofing - slowapi usa get_remote_address() que lee directamente request.client.host sin validar X-Forwarded-For contra proxies confiables. Un atacante puede enviar headers X-Forwarded-For arbitrarios y rotar IPs virtuales para evadir el límite de 10 req/min
    → Vulnerabilidad [CRITICA]: IP spoofing via X-Forwarded-For sin validación de proxy confiable
    → Vulnerabilidad [CRITICA]: CRITICAL: get_real_ip() NO implementado pero referenciado en plan - código usa get_remote_address() de slowapi que lee request.client.host sin validar X-Forwarded-For contra TRUST_PROXY_LIST
  DEUDA DE SEGURIDAD REGISTRADA:
    · [ALTA] DUPLICACIÓN DE RATE LIMITING - Existe infraestructura de rate limiting en app.py (mencionada en notas_seguridad) y ahora slowapi en /evaluar. No hay evidencia de coordinación: ¿cuál se ejecuta primero? ¿Se suman los límites o colisionan? Test C5 verifica que /health no tiene rate limit de slowapi pero no prueba interacción con sistema global
    · [ALTA] SIN VALIDACIÓN DE ORDEN DE DECORADORES - FastAPI ejecuta decoradores en orden inverso. Si @limiter.limit está después de @app.post, el rate limiting se aplica DESPUÉS de parsear JSON body con Pydantic. Payload malicioso de 100MB pasa validación de tamaño antes de rate limit → DoS bypass
    · [MEDIA] LOGS SIN HASH DE IP - Nota de seguridad dice 'IP hasheada no plaintext' pero código truncado muestra logger.info genérico. Si slowapi loggea IPs en plaintext, viola compliance de privacidad (GDPR, CCPA requieren minimización de datos)
    · [MEDIA] TEST INCOMPLETO DE RETRY-AFTER - C2 verifica que request 11 retorna 429 con X-RateLimit-Remaining: 0 pero no valida header Retry-After (RFC 6585). Cliente no sabe cuándo reintentar → UX degradada o retry storms
    · [MEDIA] SIN WHITELIST DE IPS INTERNAS - Health checks desde orquestador (:8090) o monitoreo interno consumen rate limit de /evaluar si usan misma IP pública tras NAT. Puede causar falsos positivos en alertas
    · [BAJA] DEPENDENCIAS SIN PINNING COMPLETO - requirements.txt tiene versiones exactas (fastapi==0.115.12) pero sus dependencias transitivas no están fijadas. slowapi depende de limits que depende de deprecated con vulnerabilidades conocidas en versiones antiguas
    · [BAJA] SIN MÉTRICAS DE RATE LIMITING - No hay instrumentación para ver cuántos requests son rechazados por rate limit (SLO de disponibilidad), distribución de IPs top (detección de DDoS), o tasa de falsos positivos. Debugging de incidentes es ciego
    · [ALTA] Dockerfile con --workers 1 bloquea escalado horizontal
    · [MEDIA] Sin métricas de rate limiting para detección de abuse
    · [ALTA] X-Forwarded-For sin validación de formato
    · [ALTA] Rate limiting solo en /evaluar, /orquestador sin protección
    · [MEDIA] Sin TTL en storage in-memory, IPs ocupan memoria hasta restart
    · [BAJA] Retry-After header hardcoded a 60s
    · [MEDIA] Sin documentación de limitaciones de rate limiting in-memory
    · [MEDIA] Dependencies sin version pinning exacto (usa >= en lugar de ==)
    · [ALTA] Sin validación de tamaño máximo en RequestBody de /evaluar
    · [ALTA] CORS no configurado, permite requests desde cualquier origen
    · [MEDIA] validate_trusted_proxies() solo loggea warning si TRUST_PROXY_LIST vacía pero continúa ejecución - bajo deployment real sin proxy reverso, todo request usa IP directa sin protección
    · [BAJA] Limiter inicializado sin default_limits global - cada endpoint debe decorarse manualmente, fácil olvidar endpoints futuros
    · [MEDIA] ipaddress.ip_address(ip).is_private puede lanzar ValueError si IP malformada - validate_trusted_proxies() no tiene try/except
    · [BAJA] rate_limit_handler loggea exception pero no loggea request metadata (user-agent, referer) útil para análisis forense
    · [ALTA] Criterio C3 requiere validar 8.8.8.8 como IP pública pero código solo valida .is_private - no rechaza IPs públicas explícitamente con ValueError custom
    · [MEDIA] requirements.txt sin hash pinning (==version vs ==version --hash sha256:...) - supply chain attack via PyPI compromise
    · [ALTA] /evaluar endpoint falta - tenzo_agent.py truncado en auditoría, imposible verificar @limiter.limit('10/minute') aplicado correctamente
    · [BAJA] Startup no valida que slowapi instalada correctamente - import podría fallar silenciosamente si requirements.txt no ejecutado
